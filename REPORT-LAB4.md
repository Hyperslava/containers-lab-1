### 8. Ответы на контрольные вопросы
1. В чем разница между emptyDir, hostPath и PersistentVolume? 

emptyDir - временный том, который создаётся при назначении пода на узел и существует только в течение жизненного цикла пода.

hostPath - позволяет монтировать файл или директорию с узла кластера в контейнер.

PersistentVolume - это объект, представляющий ресурс хранения в кластере Kubernetes. Он может быть как физическим, так и логическим ресурсом, например, NFS, блоком в облачном хранилище или локальным диском.

2. Как работает динамическое предоставление томов (dynamic provisioning)? 

Сначала администратор кластера определяет один или несколько объектов StorageClass, указывая provisioner и параметры для него.
Затем пользователь или разработчик создаёт PVC, указывая в нём storageClassName — имя класса хранилища, который должен использоваться.
Kubernetes использует выбранный StorageClass и соответствующий CSI-драйвер и автоматически создаёт PV в бэкенд-хранилище.
После создания PV он связывается с PVC, делая хранилище доступным для пода.

3. Что такое StorageClass и зачем он нужен? 

StorageClass - это объект, который позволяет администраторам описывать классы хранилищ, которые кластер может предлагать. Он определяет параметры динамического выделения PersistentVolumes (PV) при создании PersistentVolumeClaims (PVC).

4. Какие преимущества дает Helm по сравнению с набором YAML-файлов? 

Helm объединяет множество Kubernetes-ресурсов (поды, сервисы, конфигурации и др.) в единый пакет — чарт (chart), который можно развернуть одной командой. Каждый чарт может иметь несколько версий. Helm поддерживает отслеживание истории развёрнутых приложений (релизов) и позволяет легко откатываться к предыдущей версии при необходимости. Чарты хранятся в репозиториях, которые могут быть как публичными, так и частными. Helm позволяет добавлять различные репозитории, обновлять их и устанавливать приложения напрямую.

5. Как Helm управляет зависимостями между компонентами?

Helm управляет зависимостями между компонентами через комбинацию декларативного объявления зависимостей в файле Chart.yaml и специальных команд для работы с ними. Зависимости декларируются в секции dependencies файла Chart.yaml. В этом поле указывается список чартов, от которых зависит основной чарт, с указанием их имён, версий и репозиториев.

Helm предоставляет несколько команд для работы с зависимостями:

helm dependency list — выводит список зависимостей чарта, их версий и источников;

helm dependency update — скачивает и размещает зависимости в директорию charts/ чарта. При этом обновляется файл Chart.lock, который фиксирует точные версии всех зависимостей, обеспечивая детерминированность сборок.

helm dependency build — собирает зависимости чарта, обновляя файл charts.lock без скачивания новых пакетов. 

6. Что происходит с данными при миграции пода на другой узел в разных типах 
хранилищ? 

У временных томов жизненный цикл совпадает с жизненным циклом пода. При прекращении существования пода такой том также удаляется. Временные тома могут использоваться только одним подом.

Постоянные тома (Persistent Volumes) имеют отдельный жизненный цикл, не зависящий от пода. При миграции пода, использующего PV, на другой узел система может автоматически перемонтировать том, если это предусмотрено настройками и поддержкой со стороны поставщика хранилища.
